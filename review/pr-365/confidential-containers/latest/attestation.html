Attestation#

The NVIDIA Reference Architecture for Confidential Containers includes built-in remote attestation support for the CPU and GPU. Attestation allows a workload owner to cryptographically verify the guest Trusted Computing Base (TCB) before secrets are released to the workload.

When a workload requires a secret, such as a key to decrypt a container image or model, guest components identify the active CPU and GPU enclaves and collect hardware evidence. The evidence is sent to a remote verifier such as Trustee, which evaluates it and conditionally releases the secret.

Attestation is required for features that depend on secrets. These features include:

  • Pulling encrypted images

  • Accessing authenticated container registries

  • Using sealed secrets

  • Requesting secrets directly from workloads

To start using attestation features in your clusters, you must do the following:

  1. Provision a remote verifier/broker, like Trustee, in a trusted environment.

  2. Configure resources and policies for your attestation workflow. Exact configurations depend on your use cases.

  3. Direct your workloads to use the verifier/broker to authenticate and release secrets based on your configured policies.

Configure Remote Verifier/Broker (Trustee)#

Trustee is an open-source framework used in Confidential Containers to verify attestations and conditionally release secrets. Trustee can attest any confidential workload. For a full overview of attestation with Trustee, refer to the Trustee documentation.

To provision a Trustee instance follow the upstream Install Trustee in Docker. This is the recommended install method.

Tip

For production scenarios, Trustee should be deployed in a separate, trusted environment. For testing and development, Trustee can run in the same cluster.

After you complete installation, Trustee is configured to use the remote NVIDIA verifier, NVIDIA Remote Attestation Service (NRAS), to evaluate evidence by default.

Note

If attestation does not succeed after provisioning Trustee, enable debug logging by setting the RUST_LOG=debug environment variable in the Trustee environment. Use the Trustee log to diagnose the attestation process.

Configure Attestation Features and Workloads#

After Trustee is configured, you can begin customizing attestation workflows to enforce your desired security policies. This can include the following:

  • Configuring Trustee with the Key Broker Service (KBS) Client Tool. Refer to the upstream documentation for more information on using the KBS Client Tool.

  • Configuring policies with the Kata Containers Agent Policy documentation. You can use the genpolicy tool to autogenerate policies, or write your own manually. Both are installed with kata-deploy.

  • Configuring workloads to use attestation features such as encrypted images and authenticated container registries.

Refer to the Confidential Containers Features documentation for more information on using attestation features.
