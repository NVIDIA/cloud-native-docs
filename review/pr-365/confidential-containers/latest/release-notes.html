Release Notes#

This document describes the new features and known issues for the NVIDIA Confidential Containers Reference Architecture.

1.0.0#

This is the initial general availability (GA) release of the NVIDIA Confidential Containers Reference Architecture. The NVIDIA Confidential Containers Reference Architecture is a validated deployment model for cluster administrators who need to run GPU-accelerated AI workloads inside hardware-enforced Trusted Execution Environments (TEEs). It is designed for organizations in regulated industries that require strong isolation and cryptographic verification to protect model intellectual property and sensitive data on untrusted infrastructure. By combining NVIDIA GPU Confidential Computing capabilities with Kata Containers and the NVIDIA GPU Operator, this architecture delivers a secure, attestable, and Kubernetes-native deployment model for confidential AI workloads.

Key Features#

  • This release includes support for the following software components:

    • NVIDIA GPU Operator v26.3.1

    • NVIDIA Confidential Computing Manager for Kubernetes v0.3.0

    • NVIDIA Kata Sandbox Device Plugin v0.0.2

    • Kata Containers 3.29 (installed with the kata-deploy Helm chart)

    • Kata Lifecycle Manager 0.1.4

    • Key Broker Service (KBS) protocol 0.4.0

    • QEMU 10.1 + Patches

    • OVMF edk2-stable202511

    • Guest kernel 6.18.5

    • Containerd 2.2.2+

    • Kubernetes 1.32+

    Refer to Supported Platforms for the full list of supported hardware and software.

  • Enables GPU-accelerated workloads inside hardware-enforced Trusted Execution Environments (TEEs) using AMD SEV-SNP or Intel TDX, with kata-qemu-nvidia-gpu-snp and kata-qemu-nvidia-gpu-tdx runtime classes for secure deployment. For more information, refer to Deploy Confidential Containers.

  • Supports multi-GPU passthrough on NVSwitch-based HGX systems, including NVIDIA HGX H100 and H200 with PPCIe mode, and NVIDIA HGX B200 using NVLink encryption. For more information, refer to Deploy Confidential Containers.

  • Supports composite attestation using Trustee and the NVIDIA Remote Attestation Service (NRAS), enabling cryptographic verification of the TEE state before releasing secrets. For more information, refer to Attestation.

  • Supports model-specific GPU resource types by configuring the P_GPU_ALIAS environment variable in the Sandbox Device Plugin, enabling mixed-GPU clusters to schedule workloads to specific GPU models. For more information, refer to Configuring the Sandbox Device Plugin to Use GPU or NVSwitch Specific Resource Types.

Known Issues#

  • On single-node clusters, the Helm --wait flag may return before all kata-deploy pods finish initializing. If this occurs, wait an additional few minutes after the Helm command completes for the kata-deploy pod to reach the Running state. Refer to the known Helm issue for more information.

  • Image signature verification for signed multi-arch images is not supported.

  • PCI peer-to-peer (P2P) DMA is unsupported because IOMMUFD does not yet support mapping hardware PCI BAR regions. When a GPU is assigned as a VFIO PCI device, QEMU logs expected warnings at VM startup for each assigned device. This does not affect GPU functionality.
